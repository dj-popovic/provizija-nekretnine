# API Contract — Real Estate Website Backend (MVP)

## Document purpose

This document defines the public MVP API contract for the real-estate website backend. It is written as a human-readable contract in Markdown and serves two purposes:

1. a clear implementation guide for the backend
2. a practical integration contract between frontend and backend

This is not a full OpenAPI specification. The goal of this document is clarity, consistency, and easy implementation in Claude Code.

---

## API scope

### Included in MVP

The MVP contract includes these public endpoints:

- `GET /api/home`
- `GET /api/properties`
- `GET /api/properties/:slug-id`
- `GET /api/agents`
- `GET /api/agents/:slug`
- `GET /api/health`
- `GET /api/status`
- `POST /api/contact`
- `POST /api/inquiries/property`
- `POST /api/inquiries/agent`
- `POST /api/advertise-property`

### Explicitly out of MVP

The MVP does not include:

- authentication endpoints
- admin endpoints
- protected/internal management endpoints
- CMS-like content management endpoints
- dynamic FAQ / CTA / buying-process content endpoints
- filter facets / available-filters endpoints

---

## Response conventions

### Payload format

All request and response payloads use JSON, except standard HTTP path params and query params.

### Success response wrappers

Single-resource responses use:

```json
{
  "data": {}
}
```

Collection/list responses use:

```json
{
  "items": [],
  "pagination": {}
}
```

### Error response shape

All endpoints should use the same MVP error shape:

```json
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "Request body is invalid.",
    "details": [
      {
        "field": "email",
        "message": "Email is required."
      }
    ]
  }
}
```

#### Error fields

- `code` — stable programmatic error identifier
- `message` — short human-readable message
- `details` — optional field-level validation details

### Standard HTTP status codes for MVP

The API should document and use these standard status codes where applicable:

- `200 OK`
- `201 Created`
- `400 Bad Request`
- `404 Not Found`
- `429 Too Many Requests`
- `500 Internal Server Error`

### Response notes

For MVP, `GET /api/home` should definitely include:

- `featuredProperties`

It may also include:

- `teamPreview`

- counters
- FAQ
- buying process
- CTA blocks

### Error responses

- `500` if the backend cannot produce a valid home response

### Notes

- `teamPreview` is optional in MVP because it may remain hardcoded on the frontend.
- `featuredProperties` should be backed by a clear local/config-driven selection mechanism or another simple MVP rule.
- Image `alt` text is derived from the property title.
- `url` is a canonical relative frontend path generated by the backend using the pattern `/nekretnine/:slug-:id`.

## 2. GET /api/properties

### Purpose

Returns the public property listing with filtering, sorting, and pagination.

This is the single listing endpoint for all property browsing modes, including purchase/sale and rent flows.

### Used by page(s)

- Properties listing page
- Home page search redirect target
- Any filtered search results page

### Query/path params

Supported MVP query params:

- `transactionType`
- `propertyType`
- `location`
- `rooms`
- `priceMin`
- `priceMax`
- `areaMin`
- `areaMax`
- `page`
- `pageSize`
- `sort` *(supported in MVP, but should stay limited)*

#### Query param notes

- `transactionType` is a main filter and should be supported.
- Do not create separate endpoints for sale/rent tabs. Those are variants of this same route.
- The endpoint should not return filter metadata/facets in MVP.

#### Suggested sort values

- `newest`
- `priceAsc`
- `priceDesc`

### Request body

None.

### Success response

```json
{
  "items": [
    {
      "id": "123",
      "slug": "moderan-dvosoban-stan-liman-2",
      "url": "/nekretnine/moderan-dvosoban-stan-liman-2-123",
      "title": "Moderan dvosoban stan",
      "transactionType": "sale",
      "propertyType": "apartment",
      "location": "Novi Sad, Liman 2",
      "price": 154500,
      "images": [
        {
          "url": "https://example.com/image-1.jpg",
          "alt": "Moderan dvosoban stan"
        },
        {
          "url": "https://example.com/image-2.jpg",
          "alt": "Dnevna soba"
        }
      ],
      "area": 52,
      "rooms": 2,
      "highlights": ["Terasa", "Lift", "Parking"]
    }
  ],
  "pagination": {
    "page": 1,
    "pageSize": 12,
    "total": 245,
    "totalPages": 21
  }
}
```

### PropertyListItem fields

`PropertyListItem` should include these core MVP fields:

- `id`
- `slug`
- `url`
- `title`
- `transactionType`
- `propertyType`
- `location`
- `price`
- `images`
- `area`
- `rooms`

Optional / when available:

- `highlights`

### Image shape

The listing response should not use only `primaryImage` as a single string field.

Instead, it should use:

- `images` as an array of simple image objects

This supports slider cards on the frontend. The frontend can treat the first image as the default primary image.

Suggested simple image shape:

- `url`
- `alt`

### Response notes

The listing response should not include:

- `agentSummary`
- filter metadata / facets / available filter values

### Error responses

- `400` for invalid query params
- `500` for unexpected backend errors

### Notes

- `page` and `pageSize` should be parsed and validated on the backend.
- If `sort` is omitted, backend should apply a stable default sort, preferably `newest`.
- Image `alt` text is derived from the property title.
- `url` is a canonical relative frontend path generated by the backend using pattern `/nekretnine/:slug-:id`.
- `highlights` are derived from `<other>` XML elements (curated subset, TBD during implementation).

## 3. GET /api/properties/:slug-id

### Purpose

Returns the public detail view of a single property.

### Used by page(s)

- Property detail page

### Query/path params

#### Path params

- `slug-id`

### Request body

None.

### Route identity rule

This route should remain:

- `GET /api/properties/:slug-id`

The public URL includes both slug and ID for readability and SEO.

However:

- the property ID is the authoritative identity and lookup value
- the slug is only the SEO/canonical part of the URL

This rule must be explicit in implementation.

### Slug mismatch behavior

If the slug does not match the ID:

- backend should still resolve the property by authoritative ID
- backend should return canonical `slug` and canonical `url` in the response
- frontend may choose to correct the URL or redirect

### PropertyDetail fields

`PropertyDetail` should contain these main fields:

- `id`
- `slug`

- `images`
- `description`

- `floor`
- `heating`
- `furnished`
- `elevator`
- `registrationStatus`
- `highlights`
- `agent`
- `relatedProperties`

### Guaranteed vs optional fields

#### Guaranteed in MVP

These fields should be treated as guaranteed core detail fields:

- `id`
- `slug`
- `url`
- `title`
- `transactionType`
- `propertyType`
- `price`
- `location`

#### May be null or omitted

The following may be `null` or omitted if not available or not reliably mapped from XML:

- `images`
- `description`
- `address`
- `area`
- `rooms`
- `bathrooms`
- `floor`
- `heating`
- `furnished`
- `elevator`
- `registrationStatus`
- `highlights`
- `agent`
- `relatedProperties`

### Address rule

The API should not require a full public address by default.

`PropertyDetail` should return only the level of address/location detail intended for public display on the site.

The backend may internally preserve a richer address if available, but the public contract should expose a selective public address level.

### Missing-data behavior

For some fields, such as `furnished`, the XML may not always provide data.

In those cases:

- backend may return `null` or omit the field
- frontend may render a fallback label such as “Nema podatak”

### Related properties rule

`relatedProperties` should be embedded in the same property detail response, not exposed as a separate MVP endpoint.

### Error responses

- `404` if property with given ID cannot be resolved
- `500` for unexpected backend errors

### Notes

- `relatedProperties` should use list-item-like summary objects, not full detail objects.
- Image `alt` text is derived from the property title.
- `url` is a canonical relative frontend path generated by the backend using pattern `/nekretnine/:slug-:id`.
- `highlights` are derived from `<other>` XML elements (curated subset, TBD during implementation).
- `registrationStatus` is a three-state string derived from `<other>` XML values: `"registered"`, `"unregisterable"`, `"in_progress"`, or `null`.

# Shared conventions / common notes / naming rules

## 1. Base path

All public MVP endpoints use:

- `/api`

## 2. Resource wrappers

Use:

- `{ "data": ... }` for single-resource responses
- `{ "items": [...], "pagination": ... }` for collection responses

## 3. Pagination metadata

Collection endpoints that paginate should use:

- `page`
- `pageSize`
- `total`
- `totalPages`

## 4. Optional/null behavior

Because property data comes from an external XML source of truth, not all fields are guaranteed to exist for every record.

The contract should therefore allow some fields to be:

- `null`
- omitted

This is especially relevant for:

- optional property detail fields
- optional agent enrichment fields
- optional listing highlights
- optional media or descriptive fields

Frontend should handle missing data gracefully.

## 5. Validation documentation rule

For MVP, validation rules should be documented at the level of:

- required vs optional
- basic type expectations
- basic format expectations
- enum-like allowed values where needed

Do not over-document low-level validation constraints unless they are especially important.

## 6. Naming rules

Use normalized English field names in public API payloads where practical.

Examples:

- `transactionType`
- `propertyType`
- `priceMin`
- `priceMax`
- `areaMin`
- `areaMax`
- `fullName`

## 7. Property identity rule

The authoritative property identity is the XML `property_id`.

Public property detail route uses:

- `:slug-id`

But lookup is ID-driven, not slug-driven.

## 8. Agent identity rule

Internal XML-driven agent identity is `id_agent` (integer). In the public API, this is exposed as `agentId` (camelCase).

Public agent page route uses:

- `:slug`

For inquiry submission, the request should use:

- `agentId`

## 9. Home endpoint rule

`GET /api/home` should remain small and only include truly dynamic home-page sections in MVP.

Do not mix in static page content unless it becomes intentionally backend-driven later.

## 10. Listing endpoint rule

`GET /api/properties` is the single listing endpoint.

Do not split sale/rent/filter variants into separate routes.

Use query parameters instead.

## 11. Image rule

Where multiple property images are needed, use:

- `images` as an array of simple image objects

Do not rely only on a single `primaryImage` string if the UI expects a slider/gallery.

The internal image model uses `{ url, order }`. The API layer uses `{ url, alt }` where `alt` is derived from the property title.

## 12. Operational endpoints rule

`GET /api/health` and `GET /api/status` should remain shorter and simpler than main business endpoints, but still be documented in the same file.

## 13. Static frontend content rule

FAQ, buying-process, counters, CTA blocks, and similar static sections remain outside backend contract in MVP unless they later become dynamic.
